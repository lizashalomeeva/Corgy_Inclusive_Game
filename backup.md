# 💾 Резервне копіювання та Відновлення (Backup & Restore)

Інструкція для DevOps/Release Engineer щодо стратегії збереження критичних даних проєкту **Corgy Inclusive Game**.

## 1. Стратегія резервного копіювання

Оскільки проєкт є клієнтським мобільним додатком із Serverless API, стратегія базується на захисті вихідного коду, сертифікатів підпису та хмарному збереженні прогресу гравців.

* **Типи резервних копій:** * *Інкрементальні:* Історія комітів Git.
  * *Повні:* Резервні копії релізних збірок та криптографічних ключів.
* **Частота створення:** Безперервно (для коду через Git), одноразово (для Keystore), щомісяця (експорт логів крашів).
* **Зберігання та ротація:** Файли Keystore зберігаються у захищеному сховищі (наприклад, AWS Secrets Manager або зашифрований Google Drive) із доступом лише для Release Engineer. Ротація старих релізів в репозиторії налаштована на збереження останніх 5 версій.

## 2. Процедура резервного копіювання

* **Користувацькі дані та Бази даних:** Налаштовано механізм **Android Auto Backup**. Файли `SharedPreferences` автоматично копіюються на особистий Google Drive гравця силами ОС Android (до 25 МБ на користувача).
* **Файли конфігурацій:**
  Файл `keystore.jks` та `keystore.properties` необхідно запакувати в захищений паролем архів і завантажити у корпоративне сховище.
* **Логи системи:**
  Звіти про помилки та аналітику (Google Play Console) експортуються у форматі CSV першого числа кожного місяця для архівування.

## 3. Перевірка цілісності та Автоматизація
* *Цілісність:* Git автоматично перевіряє цілісність коду за допомогою SHA-1 хешів. Для Keystore використовується перевірка контрольними сумами.
* *Автоматизація:* Механізм GitHub Actions щоразу автоматично створює артефакти збірки при успішному злитті в гілку `master`, що слугує автоматичним бекапом готового релізу.

## 4. Процедура відновлення з резервних копій

* **Повне відновлення системи (втрата доступу до ПК розробника):**
  1. Клонувати репозиторій з GitHub на нову машину.
  2. Завантажити файл Keystore із захищеного хмарного сховища і покласти в корінь проєкту.
  3. Запустити `./docs/scripts/build_prod.bat`. Відновлення займає до 15 хвилин.
* **Вибіркове відновлення даних (помилка в коді):**
  Використання `git revert <commit_hash>` для скасування конкретної зміни без втрати загального прогресу.
* **Відновлення даних користувача:**
  Здійснюється автоматично при встановленні гри на новий телефон користувача, якщо він увійшов під своїм Google-акаунтом.
